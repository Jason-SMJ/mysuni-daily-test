# Weekly Quality Evaluation Guide

이 문서는 AI 기반 mySUNI 일일 점검 QA 애플리케이션의 운영 품질을 주간 단위로 평가하기 위한 기준과 실행 절차를 정의합니다.

목표는 아래 3가지를 분리해서 측정하는 것입니다.

- 자동화 실행이 실제로 끝까지 안정적으로 수행되는지
- LLM 최종 판정이 실제 화면 상태와 일치하는지
- Slack으로 공유된 최종 결과가 운영 판단에 사용할 수 있을 정도로 신뢰 가능한지

## 1. 평가 목적

이 애플리케이션은 Playwright 자동화, 스크린샷 캡처, LLM 판정, Slack 알림으로 구성됩니다. 따라서 품질 평가는 단순 성공률이 아니라 아래 3개 축으로 봐야 합니다.

### A. 실행 품질
- 로그인, 이동, 클릭, 팝업 전환, 캡처가 중간 실패 없이 완료되는지

### B. 판정 품질
- `정상`, `비정상`, `판단불가` 결과가 사람 검토 결과와 일치하는지

### C. 알림 품질
- Slack으로 전달된 결과가 실제 실행 결과와 동일하고 누락이 없는지

## 2. 권장 평가 기간과 방식

권장 기준은 `1주일간 Daily 배치 7회` 실행입니다.

- 운영 배치와 동일한 시간대에 매일 1회 실행
- 가능하면 실제 장애 가능성이 높은 시간대에 추가 샘플 실행 1회 수행
- `비정상`과 `판단불가`는 전수 검토
- `정상`은 최소 20% 이상 랜덤 샘플 검토

이 방식이면 운영 비용을 크게 늘리지 않으면서도 판정 신뢰도를 정량화할 수 있습니다.

## 3. 평가 단위

평가는 아래 단위로 집계합니다.

- 실행 단위: `1회 배치 실행`
- 시나리오 단위: `career_profile`, `career_recommend`, `career_mypick`, `career_1on1`, `career_myprogress`
- 항목 단위: `tests/specs/daily_check_spec.py`에 정의된 개별 체크 항목

가장 중요한 분석 단위는 `항목 단위`입니다. 동일 시나리오 안에서도 `navigate`, `click`, `popup`, `none` 유형별 난이도와 오탐 패턴이 다르기 때문입니다.

## 4. 필수 수집 데이터

주간 품질 평가를 하려면 실행 결과를 아래 필드 기준으로 남겨야 합니다.

| 필드 | 설명 |
| --- | --- |
| `run_id` | 배치 1회 실행 식별자 |
| `executed_at` | 실행 시각 |
| `scenario` | 시나리오명 |
| `item_id` | 점검 항목 번호 |
| `action_type` | `navigate`, `click`, `popup`, `none` |
| `pre_action_status` | 클릭/전환 등 사전 동작 성공 여부 |
| `screenshot_path` | 실행 스크린샷 경로 |
| `baseline_path` | 기준 이미지 경로 또는 없음 |
| `llm_result` | LLM 최종 판정 |
| `llm_reason` | 모델 응답 요약 또는 근거 |
| `human_result` | 사후 수기 검토 결과 |
| `mismatch_type` | 오탐, 미탐, 판단불가 과다 등 분류 |
| `slack_sent` | Slack 전송 성공 여부 |
| `slack_message_ref` | Slack ts, 링크, 메시지 식별자 |

현재 구조상 아래 산출물은 이미 일부 확보되어 있습니다.

- 실행 이미지: `screenshots/<service>/`
- 디버그 자료: `screenshots/debug/`
- 기준 이미지: `baselines/<service>/`

부족한 부분은 `run_id`, `human_result`, `mismatch_type`, `slack_message_ref` 같은 평가용 메타데이터입니다. 이 값은 CSV, JSONL, 또는 별도 리포트 파일로 후처리 저장하는 방식을 권장합니다.

## 5. 핵심 KPI

주간 리포트에는 아래 지표를 기본으로 포함합니다.

### 운영 안정성 지표

| 지표 | 계산 방식 | 목표 예시 |
| --- | --- | --- |
| 실행 성공률 | 성공 실행 수 / 전체 실행 수 | 98% 이상 |
| 항목 수행 성공률 | 액션 완료 항목 수 / 전체 항목 수 | 97% 이상 |
| 재시도율 | 재시도 발생 항목 수 / 전체 항목 수 | 낮을수록 좋음 |
| 플래키율 | 동일 조건 재실행 시 판정이 뒤집힌 비율 | 3% 이하 |

### 판정 정확도 지표

| 지표 | 계산 방식 | 목표 예시 |
| --- | --- | --- |
| 정확도 Accuracy | 전체 일치 수 / 전체 검토 수 | 95% 이상 |
| 비정상 Recall | 실제 비정상 중 AI가 비정상으로 탐지한 비율 | 95% 이상 |
| 비정상 Precision | AI가 비정상으로 판정한 것 중 실제 비정상 비율 | 90% 이상 |
| 판단불가율 | `판단불가` 수 / 전체 항목 수 | 5% 이하 |
| 미탐율 FN Rate | 실제 비정상인데 AI가 정상으로 본 비율 | 2% 이하 |
| 오탐율 FP Rate | 실제 정상인데 AI가 비정상으로 본 비율 | 5% 이하 |

### 알림 신뢰도 지표

| 지표 | 계산 방식 | 목표 예시 |
| --- | --- | --- |
| Slack 누락율 | Slack 미전송 건수 / 전송 대상 건수 | 0% |
| Slack 결과 일치율 | Slack 판정과 내부 결과 일치 건수 / 전체 | 100% |
| 알림 지연 | 실행 완료 후 Slack 전송까지 걸린 시간 | 5분 이내 |

## 6. 사람 검수 기준

Slack 결과가 맞는지 평가하려면 사람 검수 기준이 먼저 고정되어 있어야 합니다.

### 검수 샘플링 원칙
- `비정상`: 전수 검토
- `판단불가`: 전수 검토
- `정상`: 최소 20% 랜덤 샘플 검토
- 신규 수정이 들어간 항목: 첫 주는 전수 검토 권장

### 사람 판정 기준
- `정상`: 기대 결과가 실제 스크린샷에서 명확하게 확인됨
- `비정상`: UI 깨짐, 팝업 미노출, 전환 실패, 로딩 오류, 핵심 텍스트/영역 미표시
- `판단불가`: 스크린샷 자체가 부족하거나 가려짐, 캡처 범위가 부정확해 근거 부족

### 검수 시 참고 자료
- 실행 스크린샷
- 기준 이미지
- 디버그 JSON
- 사전 동작 성공 여부
- Slack 발송 메시지

## 7. 오탐/미탐 분류 체계

판정 불일치가 나오면 아래 중 하나로 분류합니다.

| 분류 | 설명 | 우선 조치 |
| --- | --- | --- |
| UI 변경 | 서비스 화면 구조 또는 문구 변경 | selector, baseline, 기대문구 갱신 |
| 캡처 범위 오류 | 팝업 대신 배경 화면이 찍힘 | 요소 캡처 우선순위 수정 |
| 잔존 모달 오염 | 이전 모달이 겹쳐 판정 왜곡 | 팝업 정리 로직 강화 |
| 전환 감지 실패 | popup/navigation/new-page 감지가 틀림 | Playwright 전환 감지 보강 |
| 프롬프트 모호성 | 모델이 기준을 잘못 이해 | 판정 기준 문구 구체화 |
| 기준 이미지 품질 저하 | baseline 자체가 오래되었거나 부정확 | baseline 재수집 |
| LLM 비결정성 | 동일 화면에 판정이 흔들림 | 재질의 정책 또는 보수적 판정 적용 |

현재 저장된 디버그 이력상 `career_profile` 팝업 항목은 잔존 모달 오염과 캡처 범위 오류가 실제 리스크입니다.

## 8. 1주 평가 진행 절차

### Step 1. 준비
- 배치 실행 식별용 `run_id` 부여
- 실행별 결과를 CSV 또는 JSONL로 저장
- Slack 메시지와 실행 결과를 연결할 수 있도록 메시지 식별자 기록

### Step 2. 운영 수집
- Daily 배치를 7일 연속 수행
- 실패 항목은 스크린샷과 디버그 JSON을 반드시 보존
- 변경 배포가 있었다면 날짜와 커밋을 함께 기록

### Step 3. 사람 검수
- `비정상`, `판단불가` 전건 리뷰
- `정상` 랜덤 샘플 리뷰
- 사람 기준의 최종 정답 라벨 입력

### Step 4. 지표 산출
- 시나리오별 Accuracy, Recall, Precision 계산
- 항목별 미탐, 오탐, 판단불가 다발 항목 집계
- Slack 누락 및 메시지 불일치 점검

### Step 5. 개선 우선순위 선정
- 1순위: 미탐이 발생한 항목
- 2순위: 판단불가가 반복되는 항목
- 3순위: 오탐이 반복되는 항목
- 4순위: 플래키 항목

### Step 6. 다음 주 재측정
- 동일 KPI 기준으로 개선 전후 비교
- 목표 미달 항목만 집중 모니터링

## 10. 운영 의사결정 기준

주간 평가 후 아래 기준으로 운영 상태를 판정합니다.

### Green
- 미탐율 2% 이하
- 판단불가율 5% 이하
- Slack 누락 0건

### Yellow
- 미탐율 2% 초과 5% 이하
- 특정 시나리오에서 반복 오탐 또는 판단불가 발생
- 수동 검수 보완이 필요한 수준

### Red
- 실제 장애를 정상으로 넘긴 미탐 발생
- Slack 누락 또는 잘못된 결과 전송 발생
- 동일 항목 플래키가 반복되어 자동판정 신뢰 불가

## 11. 권장 개선 우선순위

현재 구조에서는 아래 순서로 개선하는 것이 효율적입니다.

1. `career_profile` 팝업 계열 항목의 잔존 모달 제거와 요소 단위 캡처 안정화
2. `판단불가` 다발 항목의 프롬프트와 기준 이미지 정비
3. Slack 메시지와 내부 결과의 연결 키 저장
4. 주간 집계를 위한 결과 파일 포맷 표준화

## 12. 최소 운영 기준

아래 4개를 만족하면 이 QA 앱은 일일 운영 자동화 도구로 사용할 수 있습니다.

- 실제 비정상 미탐이 거의 없을 것
- Slack 결과가 내부 판정과 항상 일치할 것
- 판단불가가 운영자가 감당 가능한 수준일 것
- 동일 화면에서 판정이 자주 흔들리지 않을 것
